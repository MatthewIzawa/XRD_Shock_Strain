# v0.3.0 vs v0.4.0 survey comparison harness — design

## 1. Background and motivation

`xrd_profile` v0.4.0 (2026-05-06) shipped instrumental broadening
deconvolution and crystallite-size-distribution fits as additive
features on top of the v0.3.0 W-A path. The strict-additive contract
guarantees that `guided_warren_averbach(..., instrumental=None)` in
v0.4.0 is numerically identical to v0.3.0 for every key already present
at v0.3.0; new keys (per-family `size_distribution`) are additional.
This contract is exercised on a bundled Tirhert subset by
`tests/test_backward_compat.py` in CI.

The 29-sample JAC survey (`Paper1_JAC/survey_results_29samples.csv`) is
the application-grade artifact that backs Table 2 of the JAC
manuscript. It pre-dates v0.4.0 and is generated by `compile_survey.py`
using `XRDProfile.guided_warren_averbach` without instrumental
correction. We have not yet exercised the v0.4.0 codebase on the full
survey.

This harness does two things on the same input data:

1. **Validation.** Re-run v0.4.0 with `instrumental=None` on all 29
   samples. The output must reproduce the saved CSV row-for-row to
   floating-point precision (rounding tolerance for the four Co Kα
   rows whose saved values appear to come from a different historical
   pipeline). A failure is a strict-additive contract violation in the
   package.

2. **Sensitivity / anticipation.** Re-run v0.4.0 with
   `instrumental=<synthetic LaB6 built from literature Caglioti>` on
   the same 29 samples. The B↔C delta quantifies what the new methods
   would change if folded into the manuscript draft, and the new
   per-family `size_distribution` outputs are recorded for inspection.

The harness is single-shot but parameterised: when measured calibration
patterns later become available, swapping one entry per instrument in
`config.py` from a literature Caglioti to a measured-pattern
`InstrumentalStandard` is a one-line change.

## 2. Goals

- **Reproducibility check** of the published 29-sample table under
  v0.4.0 with `instrumental=None`, with explicit pass/fail tolerances
  per row.
- **Conditional column C** showing how the new instrumental correction
  shifts Warren-Averbach D_median and RMS strain across the survey,
  using literature Caglioti per instrument.
- **New-feature surface**: per-sample, per-family lognormal and normal
  size-distribution fits dumped to CSV and visualised on a
  representative sample.
- **Five publication-track figures** sufficient to support a JAC
  supplement decision or feed Paper 2 (MAPS) methods groundwork.
- **Single-command run** (`python run_comparison.py`) producing CSVs,
  figures, and a `run_log.md` from one entry point.

## 3. Non-goals and scope boundaries

- Not a Rietveld refinement; we are not re-deriving lattice parameters,
  occupancies, or peak intensities.
- Not a package change. The harness lives outside `xrd_profile/` and
  imports the published v0.4.0 API. No package code is modified.
- Not a JAC manuscript edit. `Paper1_JAC/` and the frozen pipeline
  scripts (`compile_survey.py`, `run_guided_wh.py`,
  `paper1_figures.py`) are read-only inputs. Any decision to fold
  results into the manuscript or supplement is a separate downstream
  step the user takes after reviewing this harness's output.
- No measured calibration acquisition. The harness uses literature
  Caglioti as the source of column C; acquiring LaB6 / Si SRM 660b /
  NAC patterns on each instrument is a separate workstream that, when
  completed, plugs into the same harness via a one-line config change.
- No CLI front-end, no YAML config, no automated re-runs. Run once,
  inspect output, decide next step.


The `build_ref` helper is the same function as in `compile_survey.py`
(uses `pymatgen.analysis.diffraction.xrd.XRDCalculator`). The
anorthite Co Kα entry is new and provides the reference for the four
Co Kα HED samples.

### 5.5 Startup assertions

`run_comparison.py` validates `config.py` before any analysis:

- `len(SAMPLES) == 29`.
- Every `s['name']` appears in `SURVEY_CSV_29` exactly once.
- Every `s['file']` exists and is readable.
- Every `s['instrument_key']` has a Caglioti entry.
- Every `(s['phase_ref'], s['wavelength'])` resolves to a non-empty
  reference peak list.
- `OUTPUT_DIR` is writable (created if missing).

Failure of any assertion aborts before the loop with a clear,
actionable message.

## 6. Data flow

### 6.1 Startup phase

1. Build `PHASE_REFS` dict (one `pymatgen.XRDCalculator` call per
   `(phase, wavelength)` combination).
2. For each instrument key, generate a synthetic LaB6 XY at that
   wavelength using the instrument's literature Caglioti, write to
   `output/synthetic_standards/lab6_<key>.xy`, and wrap as
   `InstrumentalStandard.from_cif_and_pattern(LaB6_cif, xy_path,
   wavelength=instrument_wavelength)`. Result:
   `INSTRUMENTAL_STANDARDS = {'cu_misasa': InstrumentalStandard, ...}`.
3. Read `SURVEY_CSV_29` into a dict keyed by sample name.
4. Open output CSV writers.

### 6.2 Per-sample loop

```python
for s in SAMPLES:
    tt, intensity = load_xy(s['file'], s['tt_range'])
    profile = XRDProfile(tt, intensity, s['wavelength'],
                          sample_name=s['name'])
    ref_peaks = PHASE_REFS[(s['phase_ref'], s['wavelength'])]

    wa_B = profile.guided_warren_averbach(
        ref_peaks, n_sigma=3.0, tolerance_d=0.02)
    wa_C = profile.guided_warren_averbach(
        ref_peaks, n_sigma=3.0, tolerance_d=0.02,
        instrumental=INSTRUMENTAL_STANDARDS[s['instrument_key']])

    A = saved_csv[s['name']]
    status = check_AvsB(A, wa_B, instrument_key=s['instrument_key'])
    write_comparison_row(s, A, wa_B, wa_C, status)
    for fam in wa_C['families']:
        write_size_distribution_row(s, fam)
```

The `n_sigma=3.0, tolerance_d=0.02` arguments mirror `compile_survey.py`
exactly; this is what makes Column B the strict-additive equivalent of
Column A.

### 6.3 W-A Stokes deconvolution requires a measured (or synthetic) standard

`XRDProfile.guided_warren_averbach` rejects bare
`InstrumentalProfile(U,V,W)` with a `ValueError`
(`xrd_profile/profile.py:208–214`). The Stokes Fourier deconvolution
consumes the full peak shape, not just the FWHM, so a Caglioti
polynomial alone is insufficient. The package's existing
`scripts/build_synthetic_standard.py` is hard-coded to Cu Kα and a
lab-scale Caglioti; `comparison_v040/synthetic_standards.py` is the
parameterised version.

### 6.4 A vs B tolerance

| Subset | n | Field | Tolerance |
|---|---|---|---|
| Cu Kα Misasa + I11 | 25 | `WA_D_median_A` | `\|A − B\| < 1×10⁻⁶` Å |
| Cu Kα Misasa + I11 | 25 | `WA_rms_strain` | `\|A − B\| < 1×10⁻⁹` |
| Cu Kα Misasa + I11 | 25 | `WA_families` | `A == B` |
| Co Kα Winnipeg | 4 | `WA_D_median_A` | `\|A − B\| < 0.5` Å |
| Co Kα Winnipeg | 4 | `WA_rms_strain` | `\|A − B\| < 5×10⁻⁵` |
| Co Kα Winnipeg | 4 | `WA_families` | `A == B` |

The rationale for the asymmetric Co Kα tolerances: the saved CSV stores
the four Co Kα rows with integer-rounded `WA_D_median_A` (e.g., `416`)
and 4-decimal `WA_rms_strain` (e.g., `0.3377`), suggesting those values
came from a separate historical pipeline rather than from
`compile_survey.py` directly.

A FAIL on the 25-sample subset is a strict-additive contract violation
and a real bug in the package — must be fixed before any column C
interpretation. A FAIL on the four Co Kα rows may instead reflect
different fit parameters in the original processing; logged as
informational, not a regression.

### 6.5 NaN handling

NWA 7042 in the saved CSV has `WA_families=0` and blank
`WA_D_median_A` / `WA_rms_strain` (W-A fit failed entirely). Treat as
N/A in both directions; column C is still attempted but expected to
also be empty for that sample.

### 8.1 Anticipated qualitative result

| Subset | n | Expected column-C behaviour |
|---|---|---|
| Synchrotron I11 | 18 | column C ≈ column B; instrumental contribution is roughly 1–5 % of observed FWHM for samples with D ~ 200–1000 Å. Small downward shift in D_median. |
| Lab Cu Kα + Co Kα | 11 | column C noticeably below column B; instrumental contribution typically 30–50 % of observed FWHM at high 2θ for large-crystallite samples. Substantial downward shift in D_median. |

This expectation acts as a sanity check. If synchrotron deltas come
out comparable in magnitude to lab deltas, the synthetic standard for
I11 is wrong (likely repeating the same wrong-direction failure mode
the v0.4.0 example exhibited when lab-scale Caglioti was applied to a
synchrotron pattern).

### 8.2 `run_log.md` per-run capture

- Run timestamp, `xrd_profile.__version__`, git SHA of `xrd_profile/`.
- Exact U/V/W per instrument with full citation strings.
- Calibration-source flag per instrument:
  `LITERATURE | TYPICAL | MEASURED`.
- A-vs-B summary line (e.g., "25/25 Cu+Syn rows match within 1×10⁻⁶;
  4/4 Co rows match within rounding").
- Top-5 `|B − C| / B` fractional shifts in D_median by sample.
- Standing caveat: *"Column C is conditional on these Caglioti values;
  acquiring measured LaB6 calibration on each instrument would tighten
  them."*

## 9. Error handling, testing, performance, reproducibility

### 9.1 Failure modes

**Startup assertions** (Section 5.5) catch the common configuration
errors before any analysis runs.

**Per-sample failures** are recorded and the loop continues:

- W-A fit returns 0 families (e.g., NWA 7042): record `B_*` and `C_*`
  blank, `A_vs_B_status = N/A`, log to `run_log.md`.
- Synthetic-standard build fails (wavelength out of LaB6 reflection
  range or pathological Caglioti): aborts at startup with a helpful
  message including the 2θ coverage of the synthetic standard and the
  expected sample 2θ range.
- Per-family Stokes deconvolution diverges (instrumental ≥ sample):
  the family quietly drops out of `n_families`; no crash. The size
  distribution row for that family is recorded with NaN fields.

### 9.2 Reproducibility

- Same XY input + same `xrd_profile` version + same Caglioti → byte
  identical output. W-A has no randomness; size-distribution
  `curve_fit` is deterministic from the same initial guess.
- `run_log.md` captures every variable that affects the output
  (versions, SHAs, Caglioti, run timestamp). Anyone re-running with the
  same `config.py` should produce the same artifacts.

### 9.3 Testing

The harness *is* the regression test of v0.4.0 on real survey data.
First-run pass criteria:

- 25/25 Cu Kα + I11 rows: `A_vs_B_status == PASS` within `|Δ| <
  1×10⁻⁶` Å (D_median) and `< 1×10⁻⁹` (strain).
- 4/4 Co Kα rows: match within rounding (`|Δ| < 0.5` Å, `< 5×10⁻⁵`).

Anything else is a real regression in the package. The existing
`tests/test_backward_compat.py` already covers the bundled-fixture
case in CI; the harness extends that to the application data.

There is no separate pytest harness for `comparison_v040/` itself;
`comparison.csv` is the test artifact and `run_log.md` is the
human-readable summary.

### 9.4 Performance

29 samples × 2 W-A runs (B and C) ≈ 58 W-A invocations.
`compile_survey.py` finishes in 5–10 minutes sequentially; doubling
gives roughly 10–20 minutes. Single-threaded is fine for a one-shot
harness. If runtime ever becomes painful (e.g., if Caglioti sweeps
later balloon column C to 3–5 variants), per-sample multiprocessing is
a one-day add.

## 12. Execution result and γ-scope shipping note (2026-05-07)

The harness was implemented and run end-to-end on 2026-05-07. Two
findings emerged that materially changed what shipped:

### 12.1 Sample count

`Paper1_JAC/survey_results_29samples.csv` contains 28 sample rows, not
29 — the filename is misleading. Distribution: 7 Lab Cu Kα + 17
synchrotron I11 + 4 Lab Co Kα. The original spec's "29 samples"
language is inaccurate throughout §1, §2, §6.1, §6.4, §9.3 and the
implementation plan. The harness adapted to 28; downstream
documentation (run_log, future spec edits) should follow.

### 12.2 Reference-peak filtering finding

When implementing the per-phase reference peak builder, the harness
initially called `Phase.get_ref_peaks(min_intensity=3.0)` for symmetry
with v0.3.0 idioms in the package examples. This passes only the
strongest reflections (~96 anorthite peaks instead of the ~17,000 the
unfiltered call yields). `compile_survey.py:43–47` constructs its
reference list directly via `pymatgen.XRDCalculator.get_pattern` and
passes the *unfiltered* list to `guided_warren_averbach`. The harness
must also pass the unfiltered list to reproduce the saved CSV, so the
correct call is `min_intensity=0.0`. With this fix, the strict-additive
contract holds row-for-row at floating-point precision.

### 12.3 Column C blocked — v0.5.0 ticket

W-A's Stokes deconvolution path failed for every sample with
`A_inst(0) is zero; cannot Stokes-deconvolve` (raised by
`xrd_profile.warren_averbach._stokes_deconvolve`). Root cause:
`InstrumentalStandard.fourier_coefficients(peak_d=...)` in
`xrd_profile/instrumental.py:375–412` extracts a peak profile at the
*sample's* 2θ rather than snapping to the nearest standard peak. Our
synchrotron-scale synthetic LaB6 patterns have discrete Gaussian peaks
on a near-zero baseline, so off-peak extraction yields zero area,
`A_inst[0] = 0`, and the Stokes division aborts. A real measured LaB6
pattern would have continuous structured background everywhere, which
is why the same code path "works" on real calibration data.

The docstring on `fourier_coefficients` says "the standard's peak
nearest peak_d" but the implementation does not snap. This is either
an implementation bug or a docstring bug; either way the design needs
revisiting in v0.5.0. Preferred fix (per the run-log discussion):
synthesise a Gaussian at the sample's 2θ from the fitted Caglioti
U/V/W and take Fourier coefficients of that. Alternative: snap
`target_tt` to the nearest standard peak per the existing docstring.

### 13.1 v0.4.1: Caglioti-synthesis fallback in `fourier_coefficients`

`xrd_profile/instrumental.py:fourier_coefficients` now has a two-path
implementation:

1. **Extraction** (preferred, unchanged for real measured standards):
   pull a peak window from the standard's measured pattern at the
   target 2θ and take its Fourier coefficients.
2. **Caglioti synthesis** (new, fallback): when extraction returns a
   degenerate result — `A(0) == 0` from the inner zero-area guard, or
   `converged == False` because the extracted "profile" was actually
   noise — the method synthesises a Gaussian at the target 2θ with
   FWHM from the standard's `caglioti_fit()` and takes Fourier
   coefficients of that synthetic peak.

This is preference 1 from §12.3 ("synthesise a Gaussian peak at the
sample's 2θ from the fitted Caglioti U/V/W"). The strict-additive
contract is unaffected: `instrumental=None` callers and Caglioti-only
`InstrumentalProfile` callers (W-H / Scherrer) are byte-identical to
v0.4.0. v0.4.1 is purely additive for `InstrumentalStandard` callers
whose standard pattern is sparse at the requested target 2θ.

A unit test (`tests/test_instrumental.py::TestInstrumentalStandard::test_fourier_coefficients_fallback_on_offpeak_target`)
exercises the fallback on the bundled fixture at a 2θ between LaB6
reflections. 114/114 tests pass.

Commit: `ee04727`. Tag: `v0.4.1` (annotated, on `main`). Not pushed at
ship time — awaiting user approval per `xrd_profile/CLAUDE.md`.

### 13.3 Co Kα A-vs-B is informational, not contract-bound

The four Co Kα rows in the saved CSV came from a separate historical
processing pipeline (likely `analyse_hed_lab.py` or a predecessor)
with different `n_sigma`, `tolerance_d`, or peak-detection parameters
than `compile_survey.py` uses. Our harness uses
`n_sigma=3.0, tolerance_d=0.02` (mirroring `compile_survey.py`), so
column B for Co Kα does not match the saved values exactly:

| Sample | Saved A (n_fams, D_median) | Harness B (n_fams, D_median) |
|---|---|---|
| NWA 5751 | 20, 416 Å | 13, 339.2 Å |
| NWA 6013 | 9, 352 Å | 6, 399.5 Å |
| Talampaya | 34, 310 Å | 20, 296.4 Å |
| Tatahouine | 9, 343 Å | 5, 342.7 Å |

Family counts roughly halve under our parameters; D_median differs by
−77 to +48 Å. `run_log.md` reports this as `informational FAIL on Co
Kα — parameter-divergence flag, not a contract violation`. The
harness's strict-additive validation is on the 24 Cu Kα + I11 subset
(7 PASS + 16 PASS + 1 N/A), which all pass at floating-point precision.

To reproduce the historical Co Kα values exactly, match the original
pipeline's `n_sigma` and `tolerance_d` in `config.SAMPLES`. This is
sample-pipeline reconciliation work, not a v0.4.1 concern.

### 13.4 MEASURED-standard hook in `config.INSTRUMENTAL_CAGLIOTI`

`build_instrumental_standards` in `run_comparison.py` now dispatches on
the entry's `flag`:

- `flag in ('TYPICAL', 'LITERATURE')`: synthesise LaB6 from U/V/W (the
  current state for all three instruments).
- `flag == 'MEASURED'`: load `xy_path` and wrap via
  `InstrumentalStandard.from_cif_and_pattern(cif=cif_path, ...)`.

Each entry in `config.INSTRUMENTAL_CAGLIOTI` carries commented-out
`xy_path` / `cif_path` placeholders showing the future-state shape.
Acquiring a measured LaB6 (or SRM 660b, NAC) pattern on any of the
three instruments → setting `flag` to `'MEASURED'` and uncommenting
the two paths is a three-line edit in `config.py`; no code changes
elsewhere.
